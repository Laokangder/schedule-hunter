# 全栈功能闭环实现差异报告（2026-04-26）

## 一、数据解析层（LLM Service）

### 痛点：Markdown/非结构化导致解析崩溃

LLM 返回内容可能包含思考链（`<think>...</think>`）、markdown 代码块、多余逗号、截断 JSON 等不可控格式。原 `json.loads()` 直接解析在 LLM 返回"裸奔 JSON"或混合格式时频繁抛异常。

### 变更：暴力 `{}` 截断 + 强约束 Prompt

**llm_service.py** 核心改动：

```
# 原实现 — 脆弱
raw_content = response_data["choices"][0]["message"]["content"]
parsed = json.loads(raw_content)   # 遇 markdown/截断/think 必崩

# 新实现 — 暴力截断
raw_content = re.sub(r'<think>.*?</think>', '', raw_content, flags=re.DOTALL)
start_idx = raw_content.find('{')
end_idx = raw_content.rfind('}')
raw_content = raw_content[start_idx:end_idx+1]
parsed = json.loads(raw_content)
```

**System Prompt 硬化** — 新增 2 条 Few-shot 示例 + location 强制提取指令：

```
1. 输入"明天下午3点去望京SOHO找张总谈合作"
   输出 {"title":"谈合作","start_time":"2026-04-26T15:00:00+08:00","end_time":"...","location":"望京SOHO","confidence":0.95}

2. 输入"在和平饭店吃饭"
   输出 {"title":"吃饭","start_time":"2026-04-25T12:00:00+08:00","end_time":"...","location":"和平饭店","confidence":0.95}
```

### 效果

| 场景 | 原行为 | 新行为 |
|------|--------|--------|
| LLM 返回 `<think>...</think>{\"title\":\"吃饭\"}` | json.loads 异常 → HTTP 500 | 正则剔除 think 块 → 正常解析 |
| 输入"在和平饭店吃饭" | title="在和平饭店吃饭", location=null | title="吃饭", location="和平饭店" |
| 输入"明天去望京SOHO谈合作" | title="明天去望京SOHO谈合作" | title="谈合作", location="望京SOHO" |
| LLM 返回截断 JSON | json.decoder.JSONDecodeError | `rfind('}')` 兜底截取 |

## 二、数据模型层（Backend Models）

### 痛点：Pydantic 校验空值引发 422

前端或 LLM 返回的 `end_time`、`location` 为 `null` 时，Pydantic 模型因字段类型为非 Optional 抛出验证错误；`ConflictItem.detail` 缺失时同样 422。

### 变更：Optional 化 + 后端前置补全

**request.py** 模型调整：

| 模型 | 字段 | 原类型 | 新类型 | 说明 |
|------|------|--------|--------|------|
| ParsedTask | `end_time` | `str` | `Optional[str]` | LLM 可返回值 null |
| ParsedTask | `location` | `str` | `Optional[str]` | 无地点时数据库存 null |
| CandidateTask | `end_time` | `str` | `Optional[str]` | 冲突检查时兼容 |
| CandidateTask | `location` | `str` | `Optional[str]` | 同上 |

**response.py** 模型调整：

| 模型 | 字段 | 原类型 | 新类型 | 说明 |
|------|------|--------|--------|------|
| ConflictItem | `detail` | `str` | `Optional[str] = None` | 无冲突详情时不抛验证错误 |

**parse_service.py** 前置补全逻辑：

```python
if llm_result.get("end_time") is None:
    if llm_result.get("start_time"):
        start_dt = datetime.fromisoformat(llm_result["start_time"])
        if start_dt.tzinfo is None:
            start_dt = start_dt.replace(tzinfo=self.timezone)
        llm_result["end_time"] = (start_dt + timedelta(minutes=60)).isoformat()
```

时区安全处理：无时区字符串不再被当作 UTC，强制挂载 `Asia/Shanghai`。

### 效果

| 场景 | 原行为 | 新行为 |
|------|--------|--------|
| `end_time: null` | Pydantic 校验 422 | Optional 接受 → 自动推算 60 分钟 |
| `location: null` | 校验失败 | Optional 接受 → 数据库存 null |
| `ConflictItem.detail` 为空 | 422 | 默认 `None` 正常序列化 |
| 时区裸字符串 `2026-04-26T00:11:00` | 当作 UTC → 偏移 +8h | 原地挂载 +08:00 → 零偏移 |

## 三、状态管理层（Frontend Store）

### 痛点：`islandState` vs `island_state` 命名混乱 + `is_loading` 永不重置

ScheduleIsland.vue 中曾使用 `store.islandState`（驼峰）引用 store 中实际为 `island_state`（下划线）的字段，导致 `v-if="isTracking/computed"` 永远为 false，灵动岛停滞在 idle 态。同时 `handle_native_sniff` 未实现 `try...finally`，loading 状态在异常分支未被复位 → "解析中..."按钮无限转圈。

### 变更：命名统一 + try...finally 硬复位

**ScheduleIsland.vue** 全部状态引用改为下划线：

```
// 旧（失效引用）
store.islandState
store.isConflict
store.conflictDetail

// 新（有效引用）
store.island_state
store.is_conflict
store.conflict_detail
```

**useTaskStore.js `handle_native_sniff`** 结构改造：

```javascript
async handle_native_sniff(source_text, input_type = 'manual') {
  this.is_loading = true    // 进入立即锁住
  try {
    // ... API 调用 + 状态机转换 ...
    return { success: true, data: raw_data }
  } catch (error) {
    this.error = error.message
    return { success: false, error: error.message }
  } finally {
    this.is_loading = false  // 任何出口都复位
  }
}
```

### 效果

| 场景 | 原行为 | 新行为 |
|------|--------|--------|
| 解析按钮点击 | 无限转圈 | 成功/异常后立即恢复"智能解析"文字 |
| 灵动岛提醒态 | 按钮无响应 | 确认/取消/推迟正常触发 |
| 冲突预警按钮 | 点击无效果 | 忽略/调整/确认创建均能驱动状态机 |
| 倒计时态展开 | store.islandState 永为 undefined | 正确响应 tracking 状态 |

## 四、业务闭环层（Automation & CRUD）

### 痛点：任务创建后不刷新、日历点击无联动、标题"返祖"

原来 `create_task_from_active` 完成后仅 `this.tasks.push(created_task)`，不重新请求后端；日历日期点击后列表不切换；确认创建时 `title` 字段回退为原始输入文本（如"在和平饭店吃饭"）。

### 变更：全链路闭环链路贯通

**`create_task_from_active` 确认闭环**：

```
用户点击灵动岛"确认"
  → POST /api/v1/tasks (body.parsed.title = active_task.title)
  → 后端写入数据库
  → clear_active_task() → island_state = 'idle'
  → fetch_latest_tasks() 携带 this.selected_date
  → GET /api/v1/tasks?date=2026-04-27
  → today_tasks 按 selected_date 过滤
  → BountyList 渲染新列表
```

**`selected_date` 驱动日历联动**：

```
state.selected_date = new Date().toISOString().split('T')[0]  // 初始化今天

CalendarGrid @click → store.setSelectedDate("2026-04-27")
  → this.selected_date = "2026-04-27"
  → fetch_latest_tasks() 使用 this.selected_date 请求
  → today_tasks getter 过滤条件: task_date === store.selected_date
  → 列表刷新 + <h2>{{ pageTitle }}</h2> 动态标题
```

**标题纯度锁定 — `create_task_from_active`**：

```javascript
// 旧: title: this.active_task.title || this.captured_text || ''
// 新: title: this.active_task.title
// captured_text 仅保留在 source_text 字段，永不污染入库 title
```

### 效果

| 交互 | 原行为 | 新行为 |
|------|--------|--------|
| "确认"创建任务 | 列表不刷新，需 F5 | 自动 refetch → 列表实时更新 |
| 日历点击 4/27 | 列表不变 | GET ?date=2026-04-27 → 显示当天任务 |
| 选中非今天 | 标题始终"今日悬赏" | "2026-04-27 悬赏" 动态切换 |
| 确认创建"在和平饭店吃饭" | title="在和平饭店吃饭" | title="吃饭" (纯净 AI 标题) |

## 五、遗留问题与风险

### 时区偏移（深夜跨天查询）

**场景**：用户在 `2026-04-25 23:55:00 +08:00` 创建 start_time=`2026-04-26T00:30:00+08:00` 的任务。前端 `today_tasks` 按 `start_time.split('T')[0]` 取日期 = `2026-04-26`。后端 `GET /tasks?date=2026-04-26` 使用 `LIKE '2026-04-26%'` 匹配。此链路已正常工作。

**已知边界**：当 LLM 返回时区裸字符串（如 `2026-04-26T00:11:00` 无 `+08:00`）时，`datetime.fromisoformat()` 产生 naive datetime；已通过 `if start_dt.tzinfo is None: start_dt = start_dt.replace(tzinfo=self.timezone)` 修复。

**潜在优化点**：
- 统一前端/后端/LLM 三方的时区协议，要求在 API 契约中强制 timezone 字段
- 考虑 UTC 存储 + 本地化展示的架构方案，避免时区边界值问题
- `Date()` 跨天临界点的毫秒级精度可能导致展示偏差（23:59:59 vs 00:00:00）

### 缺失模块（未实现）

| 模块 | 影响 | 优先级 |
|------|------|--------|
| apscheduler 定时提醒 | 无法主动推送30分钟提醒 | 高 |
| DELETE /api/v1/tasks/{id} | 无法删除已有任务 | 中 |
| Android 原生嗅探 | 无法捕获系统通知/剪贴板 | 高 |
| BlueLM 真实 API 接入 | 返回模拟数据 | 中 |
| WebSocket 主动推送 | 后端无法主动通知前端 | 中 |

### 风险评级

| 风险 | 等级 | 缓释方案 |
|------|------|----------|
| LLM 返回空/无效 JSON | 中 | 正则 fallback + `{}/{rfind}` 双保险 |
| 跨天时区边界值 | 低 | 已修复 naive datetime 挂载时区 |
| 灵动岛状态机死锁 | 低 | try...finally 硬复位 + clearActiveTask 全量重置 |
| Pydantic 模型与前/后约定不一致 | 中 | `model_dump()` 导出，减少字段耦合 |
