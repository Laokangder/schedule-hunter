## 2. 健壮性与容错 (Robustness)

### 2.1 异常处理对比

| 维度 | 架构要求 | 现状 | 差距 |
|------|----------|------|------|
| LLM 调用降级 | fallback 正则 | ✅ 实现 | — |
| JSON 解析保护 | 未明确要求 | ✅ 暴力 {} 截断 | 超预期 |
| 网络重试 (指数退避) | 最大 3 次,幂等 | ❌ 无 | 前端无重试,后端无幂等 |
| 本地缓存 (SQLite) | 离线缓存待解析文本 | ❌ 无 | 离线完全不可用 |
| 错误码体系 | 8 个业务错误码 (1001-5000) | ⚠️ 后端返回,前端未消费 | 前端不处理错误码,仅显示通用错误 |
| ai_fallback 标记 | BlueLM 降级时标记 | ❌ 无 | 无法区分真实 AI 与 fallback 结果 |
| 用户感知降级 | 灵动岛/Toast 告知 | ❌ 无 | 降级静默发生 |

### 2.2 当前代码中的风险点

**后端风险**

```
llm_service.py: parse_task()
  → response.raise_for_status() 未捕获 httpx.TimeoutException
  → 超时场景直接 500,用户无法重试

parse_service.py: parse()
  → await self.llm_service.parse_task() 无超时兜底
  → LLM 卡死 30s 时前端无反馈

task_repo.py: get_tasks_by_date()
  → SQL 使用 LIKE 字符串拼接 (无注入风险,但有隐式类型转换)
  → date=None 场景返回全部任务,大数据集时无分页

task.py: create_task()
  → 无 request_id 幂等校验
  → 网络重试 → 重复创建 → 数据库增长
```

**前端风险**

```
useTaskStore.js: tasks 数组只增不减
  → 无数据淘汰策略 → 内存泄漏
  → 日历点击联动仅查询当天,但 store.tasks 积累全部数据

apiService.js: 无超时配置
  → axios 默认无限等待
  → LLM 解析 30s+ 时前端 is_loading 死锁

ScheduleIsland.vue: noop handler 危险
  → handleReschedule() 调用 store.clearActiveTask()
  → 用户点击"调整时间"实际效果是取消,无反馈
```

**Priority: P1** → 网络重试、错误码消费、超时保护需在下次迭代补齐

## 3. 性能与可扩展性 (Scalability)

### 3.1 当前架构瓶颈

**数据库层**

```
SQLite (单文件)
  → 无并发写入保护 (FastAPI 异步 + SQLite 单线程 = 锁竞争)
  → 无连接池复用
  → GET /api/v1/tasks?date= 全表扫描 (无索引)
```

当前 `task_repo.py` 查询日志验证：
```
查询到5个任务  (单次扫描全表)
查询到13个任务 (单次扫描全表)
```

**API 层**

```
FastAPI 单进程单线程:
  → main.py 无 gunicorn/uvicorn workers 配置
  → BlueLM httpx 调用阻塞事件循环
  → 1 个慢请求阻塞所有请求 (head-of-line blocking)
```

**前端状态管理**

```
store.tasks 全局数组:
  → 无分页/懒加载逻辑
  → 日历标记去重 (marked_dates getter) 全部遍历
  → 用户有 10000+ 任务时性能退化
```

### 3.2 架构设计与当前实现差距

| 维度 | 架构目标 | 现状 | 差距分析 |
|------|----------|------|----------|
| 并发能力 | QPS ≥ 100 | 单线程 SQLite ≈ 10 QPS | SQLite → PostgreSQL, 加 uvicorn workers |
| 响应时间 | LLM 解析 ≤ 1.5s | 同步阻塞 LLM (30s+) | 需异步 LLM 调用 + 超时熔断 |
| 前端离线 | 本地 SQLite 缓存 | 无 | 首次加载全部依赖网络 |
| 数据分片 | 按日期分区 | 无索引全表扫描 | task_repo.py 加 start_time 索引 |
| 水平扩展 | Docker + K8s | 单机 single process | 无容器化 |

### 3.3 内存与数据增长预测

```
当前: 5-15 条任务, 全表扫描 < 1ms

1 个月: ~300 条
  全表扫描 ≈ 3ms  (可接受)
  前端 store.tasks: ~100KB (可接受)

6 个月: ~1800 条
  全表扫描 ≈ 18ms (开始显著)
  前端 store.tasks: ~600KB (开始影响低端设备)

12 个月: ~3600 条
  全表扫描 ≈ 36ms (影响用户体验)
  前端 store.tasks: >1MB (需要分页)
```

**Priority: P2** → 立即需要数据库索引,3 个月内需要分页和后端定时清理

| 架构规范 | 实现状态 | 差距 |
|----------|----------|------|
| 静默态点击展开最近 3 条日程 | ❌ 未实现 | 无操作 |
| 倒计时态长按跳转地图/链接 | ❌ 未实现 | 无响应 |
| 提醒态滑动切换下一条任务 | ❌ 未实现 | 单条展示,无手势 |
| 冲突预警态 AI 建议排序 | ⚠️ 展示建议列表,未排序 | 后端返回顺序固定 |
| 双击确认"已知晓" | ❌ 未实现 | 仅点击取消 |
| 下拉收起灵动岛 | ❌ 未实现 | 固定展示,无法收起 |

### 4.2 视觉与反馈缺失

| 交互场景 | 当前行为 | 理想行为 | 差距 |
|----------|----------|----------|------|
| 解析成功 | 灵动岛从 idle → reminder | 对应态动画过渡 | 无 Transition 名称 |
| 解析失败 | 不显示 | Toast 提示 + 错误详情 | 用户无感知 |
| 网络错误 | error 记录在 store | 灵动岛显示"网络异常"卡片 | 无前端反馈 |
| 任务创建确认 | 自动消失 | 成功动画 + 列表滚动到新任务 | 无操作反馈 |
| 冲突解决 | 调用 clearActiveTask | 平滑过渡到 track/reminder | 突兀切换 |

### 4.3 降级交互

| 架构规范 | 实现状态 | 差距 |
|----------|----------|------|
| 权限缺失引导入口 | ❌ 无 | App 无降级概念,浏览器无权限场景 |
| 离线缓存与重试 | ❌ 无 | 网络中断完全不可用 |
| 输入法快捷入口 | ❌ 无 | 仅一个 textarea |

### 4.4 数据展示质量

| 字段 | 当前表现 | 理想表现 | 差距 |
|------|----------|----------|------|
| 时间展示 | `new Date(isoString).toLocaleString()` | 口语化"今天 15:00" "明天 09:00" | 无相对时间转换 |
| 地点为空 | 显示"未设置地点" | 隐藏地点行 | 空信息占据空间 |
| 参与者解析 | 存数组,不展示 | 标签化"张总" | 数据有,不展示 |
| 置信度 | 存在,不展示 | 低 < 0.5 时显示"AI 觉得不太确定" | 用户无法感知可靠性 |

**Priority: P1** → 错误反馈和过渡动画直接影响用户对 App 可靠性的判断
