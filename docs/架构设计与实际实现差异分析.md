## 二、API 接口契约差异

### 2.1 接口覆盖对比

| 架构定义接口 | 前端实现 | 差异说明 |
|-------------|----------|----------|
| POST /api/v1/task/parse | ✅ 已实现 | 请求/响应字段基本一致 |
| POST /api/v1/tasks | ✅ 已实现 | 字段映射略有差异 |
| GET /api/v1/tasks | ✅ 已实现 | 响应结构一致 |
| POST /api/v1/tasks/conflict-check | ✅ 已实现 | 字段名称略有差异 |
| WebSocket /ws/tasks | ⚠️ 框架存在 | 连接代码存在但未完整实现 |

### 2.2 POST /api/v1/task/parse 响应字段差异

**架构设计响应：**
```json
{
  "code": 0,
  "message": "parsed",
  "data": {
    "parsed": { ... },
    "is_conflict": true,
    "conflict": { ... },
    "suggestions": [...],
    "needs_confirmation": false,
    "ambiguities": []
  },
  "trace_id": "..."
}
```

**实际响应处理（useTaskStore.js）：**
- 响应结构一致
- `data.is_conflict` 字段处理正确
- `data.conflict` 字段处理正确
- `data.suggestions` 映射为 `conflict_suggestions` 存储
- `data.needs_confirmation` 和 `data.ambiguities` 处理正确

### 2.3 POST /api/v1/tasks/conflict-check 字段差异

**架构设计：**
```json
{
  "has_conflict": true,
  "conflicts": [...],
  "ai_summary": "..."
}
```

**实际实现差异：**
- `has_conflict` 字段未在请求体中出现（架构要求）
- `scope` 对象中 `check_type` 默认值一致
- `scope` 对象中 `include_travel_time` 和 `travel_buffer_minutes` 一致

### 3.3 UI 状态机 ⚠️ 部分实现

**架构定义四态：**

| 状态 | 架构定义 | 实际实现 |
|------|----------|----------|
| 静默 (Silent) | 胶囊最小化、低对比度中性色 | ✅ ScheduleIsland idle 状态 |
| 倒计时 (Countdown) | 亮度递增、显示剩余时间、呼吸动画 | ⚠️ tracking 状态，动画简化 |
| 任务提醒 (Reminder) | 展开半卡片、三段式操作 | ✅ reminder 状态 |
| 冲突预警 (Conflict Alert) | 红色警示、显示冲突摘要 | ⚠️ warning 状态，但交互简化 |

**状态切换规则实现差异：**

架构设计：
```
静默 ──存在临近任务──▶ 倒计时 ──进入提醒阈值──▶ 任务提醒
  ↑                              │
  │                              ↓
  └──────用户修正完成─────── 冲突预警
```

实际实现（useTaskStore.js）：
- `_schedule_reminder` 方法实现了部分状态流转
- 但状态命名使用 `islandState`（大写），与 store 定义 `island_state`（小写）不一致
- ScheduleIsland 组件中使用了 `store.islandState`（大写），与 store 定义冲突

### 3.4 WebSocket 连接 ⚠️ 框架存在

**架构定义推送消息类型：**
```json
{
  "type": "reminder_triggered | conflict_detected | task_updated",
  "task_id": "...",
  "timestamp": "...",
  "payload": { ... }
}
```

**实际实现（useTaskStore.js）：**
- `connect_web_socket()` 方法存在
- 处理了 `schedule_sniff`, `reminder_triggered`, `conflict_detected` 三种消息类型
- 但未处理 `task_updated` 类型
- 断线重连逻辑存在（5秒重连）
- 实际使用场景有限（未主动触发）

## 五、错误处理差异

### 5.1 错误码体系 ❌ 未实现

架构定义的错误码：

| 错误码 | 含义 | 前端处理建议 |
|--------|------|-------------|
| 1001 | 解析文本为空 | 提示用户输入日程内容 |
| 1002 | 时间表达式无法识别 | 提示用户补充具体时间 |
| 1003 | 解析置信度过低（< 0.6） | 弹出确认界面 |
| 2001 | 任务时间无效 | 提示用户检查时间 |
| 2002 | 冲突检测失败 | 降级为无冲突模式 |
| 2003 | 冲突等级过高 | 引导用户决策 |
| 3001 | 任务写入失败 | 提示重试 |
| 5000 | 服务器内部错误 | 显示服务暂不可用 |

**实际实现：**
- apiService.js 仅抛出通用 Error
- mockProvider.js 无错误模拟
- useTaskStore.js 使用 `this.error` 存储字符串错误信息
- 无错误码解析与用户友好提示

### 5.2 网络异常预案 ❌ 未实现

| 异常类型 | 架构要求 | 实际实现 |
|----------|----------|----------|
| 网络中断 | 本地缓存待解析文本 | ❌ 无 |
| 网络恢复 | 自动重试（指数退避） | ❌ 无 |
| 服务端 5xx | 显示提示允许重试 | ⚠️ 仅 console.error |
| BlueLM 降级 | 返回 ai_fallback: true | ❌ 无 |

## 六、代码质量问题

### 6.1 状态命名不一致

**store 定义（useTaskStore.js）：**
```javascript
island_state: 'idle',  // 下划线命名
is_conflict: false,
conflict_detail: '',
```

**组件使用（ScheduleIsland.vue）：**
```javascript
store.islandState  // 驼峰命名 - 错误
store.isConflict   // 驼峰命名 - 错误
store.conflictDetail // 驼峰命名 - 错误
```

**组件使用（Index.vue）：**
```javascript
store.islandState = 'tracking'  // 错误
```

**组件使用（AddTaskModal.vue）：**
```javascript
store.setActiveTask(task)  // 错误，应为 set_active_task
store.islandState = 'tracking'  // 错误
```

### 6.2 getter 命名不匹配

**store 定义：**
```javascript
getters: {
  is_idle: (state) => state.island_state === 'idle',
  is_tracking: (state) => state.island_state === 'tracking',
  is_reminder: (state) => state.island_state === 'reminder',
  is_warning: (state) => state.island_state === 'warning',
}
```

**组件使用（ScheduleIsland.vue）：**
```javascript
store.isIdle     // 正确
store.isTracking // 正确
store.isWarning // 正确
store.isReminder // 正确
```
