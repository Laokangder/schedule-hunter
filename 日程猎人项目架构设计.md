# 日程猎人（Schedule Hunter）| 架构计划与开发契约书

## 0. 项目定位

**项目名称：** 日程猎人（Schedule Hunter）
**项目目标：** 基于“大前端系统级嗅探 + 本地高算力大模型”的日程调度工具。利用跨平台框架打通 Android 底层权限（通知监听、无障碍、Intent 分享），经 FastAPI 后端接入蓝心大模型（BlueLM）进行自然语言意图提取与时间冲突检测，以“灵动岛”形式输出智能调度结果。
**设计原则：**

- 系统级嗅探优先：自动捕获通知、剪贴板、分享 Intent 中的日程意图
- 本地算力优先：RTX 5060 执行高频解析、规则匹配与局部推理
- 大模型增强：BlueLM 辅助歧义消解、补全信息与冲突解释
- 冲突主动预警：在任务创建前执行日程冲突判定
- 移动端低打扰：灵动岛形式输出状态，不抢占主内容区域
- 绝对执行纪律：契约锁定后禁止擅自修改接口字段，前后端并行开发以契约 Mock 为准

## 1. 前后端边界与职责划分

### 1.1 物理边界

| 层级 | 技术栈 | 部署位置 | 负责边界 |
|------|--------|----------|----------|
| 大前端（客户端） | Vue 3 + Tailwind CSS + Uni-app / Capacitor | Android 设备 | 系统权限申请、文本监听、UI 状态机、发起请求 |
| 后端（服务端） | Python FastAPI + BlueLM | RTX 5060 服务器 | 接收原始文本、LLM 解析、冲突算法、返回标准化 JSON |

### 1.2 前端职责清单（Device Side）

- **权限管理**：申请并维护 Android 高危权限（无障碍 AccessibilityService、通知监听 NotificationListenerService、前台服务 ForegroundService）
- **系统文本监听**：截取系统剪贴板变化、监控通知栏文本、捕获分享 Intent 传入数据
- **UI 状态机**：四态（静默 / 倒计时 / 提醒 / 冲突预警）的渲染与动画逻辑
- **HTTP 请求**：封装 REST 客户端，发起 `parse` / `tasks` / `conflict-check` 请求
- **WebSocket 连接**：建立长连接，接收服务端实时推送并驱动灵动岛状态更新
- **降级交互兜底**：权限缺失时提供手动输入 / 分享 Intent 入口，保证最低可用性

### 1.3 后端职责清单（Server Side）

- **文本解析入口**：接收前端原始文本（`source_text`），不信任任何客户端预解析结果
- **时间表达式解析**：本地正则 + rule-based 引擎执行第一轮时间提取
- **BlueLM 调用**：执行意图提取、歧义消解、信息补全
- **冲突检测算法**：时间重叠（time_overlap）、出行冲突（travel_conflict）、优先级冲突（priority_conflict）
- **标准化响应**：返回包含解析结果、冲突标识、调度建议的 JSON 包
- **会话上下文维护**：存储近期任务上下文，支持多轮追问
- **WebSocket 实时推送**：提醒触发时主动推送到客户端
- **全链路追踪**：每个请求分配 `trace_id`，串联请求-响应-日志

### 1.4 边界协议要点

- 前端**不解析时间**，仅做原始文本透传
- 后端**不维护 UI 状态**，只返回业务语义数据
- 双方以 `application/json` 为唯一数据格式
- 所有接口均需 `trace_id` 贯穿请求与响应
- 接口契约锁定后，字段增删需双方 Tech Lead 评审方可变更

## 2. 核心 API 契约设计

### 2.1 接口总览

| 方法 | 路径 | 用途 |
|------|------|------|
| POST | `/api/v1/task/parse` | 核心接口：解析嗅探文本，输出结构化任务对象 |
| POST | `/api/v1/tasks` | 确认创建任务 |
| GET | `/api/v1/tasks` | 按日期查询任务列表 |
| POST | `/api/v1/tasks/conflict-check` | 主动发起冲突检测 |
| WebSocket | `/ws/tasks` | 实时任务状态推送 |

---

### 2.2 核心接口：POST /api/v1/task/parse

**接口描述：** 接收前端嗅探到的原始文本，经后端 LLM 解析后返回标准化任务对象与冲突状态。前端灵动岛根据响应更新 UI 状态。

**请求体：**

```json
{
  "request_id": "req_20260420_0001",
  "source_text": "明天下午三点去望京 SOHO 找张总谈合作",
  "context": {
    "recent_tasks": [],
    "user_preferences": {
      "default_duration_minutes": 60,
      "timezone": "Asia/Shanghai"
    }
  },
  "meta": {
    "input_type": "notification | clipboard | intent | manual",
    "client_timestamp": "2026-04-20T16:30:00+08:00",
    "app_version": "1.0.0"
  }
}
```

**字段说明：**

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| request_id | string | 是 | 客户端生成的唯一请求标识 |
| source_text | string | 是 | 原始待解析文本，最大长度 500 字符 |
| context.recent_tasks | array | 否 | 近期任务上下文，供 LLM 消歧使用 |
| context.user_preferences | object | 否 | 用户偏好，包含默认时长与时区 |
| meta.input_type | enum | 是 | 输入类型枚举：notification / clipboard / intent / manual |
| meta.client_timestamp | string | 是 | ISO 8601 格式，客户端时间戳 |
| meta.app_version | string | 否 | 客户端版本号 |

**响应体：**

```json
{
  "code": 0,
  "message": "parsed",
  "data": {
    "parsed": {
      "title": "与张总在望京 SOHO 谈合作",
      "start_time": "2026-04-21T15:00:00+08:00",
      "end_time": "2026-04-21T16:00:00+08:00",
      "location": "望京 SOHO",
      "participants": ["张总"],
      "timezone": "Asia/Shanghai",
      "confidence": 0.94
    },
    "is_conflict": true,
    "conflict": {
      "conflict_type": "time_overlap",
      "conflict_level": "medium",
      "conflicting_task_id": "task_991200",
      "detail": "15:00-16:00 与「产品评审会」时间重叠"
    },
    "suggestions": [
      {
        "action": "reschedule",
        "suggested_time": "2026-04-21T16:30:00+08:00",
        "description": "推迟 1.5 小时避免重叠"
      },
      {
        "action": "shorten",
        "suggested_duration_minutes": 45,
        "description": "压缩至 45 分钟，留出缓冲"
      },
      {
        "action": "dismiss",
        "description": "忽略此次检测，不创建任务"
      }
    ],
    "needs_confirmation": false,
    "ambiguities": []
  },
  "trace_id": "trace_20260420_aaa001"
}
```

**响应字段说明：**

| 字段 | 类型 | 说明 |
|------|------|------|
| code | integer | 业务状态码，0=成功，非0=失败 |
| message | string | 状态描述 |
| data.parsed | object | 解析后的结构化任务对象 |
| data.parsed.title | string | 事件名称 |
| data.parsed.start_time | string | ISO 8601 格式，开始时间 |
| data.parsed.end_time | string | ISO 8601 格式，结束时间 |
| data.parsed.confidence | float | 解析置信度，范围 0.0~1.0 |
| data.parsed.location | string | 地点，为空时为 null |
| data.parsed.participants | array | 参与人员列表 |
| data.is_conflict | boolean | 是否存在冲突 |
| data.conflict | object | 冲突详情，无冲突时为 null |
| data.conflict.conflict_type | enum | 冲突类型：time_overlap / travel_conflict / priority_conflict / semantic_conflict |
| data.conflict.conflict_level | enum | 冲突等级：low / medium / high |
| data.suggestions | array | AI 调度建议列表 |
| data.needs_confirmation | boolean | 是否需要用户确认歧义 |
| data.ambiguities | array | 歧义字段列表，为空时为 [] |
| trace_id | string | 全链路追踪 ID |

### 2.7 错误码体系

| 错误码 | 含义 | 前端处理建议 |
|--------|------|-------------|
| 0 | 成功 | — |
| 1001 | 解析文本为空 | 提示用户输入日程内容 |
| 1002 | 时间表达式无法识别 | 提示用户补充具体时间 |
| 1003 | 解析置信度过低（< 0.6） | 弹出确认界面，让用户核对 |
| 2001 | 任务时间无效 | 提示用户检查时间是否合理 |
| 2002 | 冲突检测失败 | 降级为无冲突模式，允许创建 |
| 2003 | 冲突等级过高需用户确认 | 展示冲突详情，引导用户决策 |
| 3001 | 任务写入失败 | 提示重试 |
| 5000 | 服务器内部错误 | 显示「服务暂不可用」，允许重试 |

## 3. 标准化开发工作流


### Phase 1：契约锁定（预计 1 周）

**目标：** 前后端完全解耦，以 Mock 数据为唯一事实来源，独立推进 UI 与 AI 逻辑。

**前端行动项：**
- 基于设计稿实现灵动岛四态 UI（静默 / 倒计时 / 提醒 / 冲突预警）
- 使用 Mock JSON 渲染所有状态，验证 UI 组件可用性
- 封装 HTTP 客户端，基于契约 Schema mock 所有接口响应
- 建立 WebSocket 模拟连接，验证前端状态机逻辑

**后端行动项：**
- 搭建 FastAPI 项目骨架（路由 / 中间件 / 日志 / 错误处理）
- 接入 BlueLM（蓝心大模型），调通 Prompt → 结构化 JSON 输出链路
- 硬编码冲突判定逻辑（时间重叠优先实现），输出标准化响应
- 部署 `trace_id` 中间件与结构化日志基础设施
- 提供 Swagger/OpenAPI 在线文档，作为前后端唯一事实来源

**契约管理约定：**
- Mock 数据路径：`packages/api-contracts/mocks/`
- API 版本前缀：`/api/v1/`
- 契约变更必须提交 PR，由双方 Tech Lead 评审
- 禁止前端在 Mock 阶段自行修改字段名或类型

---

### Phase 2：核心链路攻坚（预计 2 周）

**目标：** 前端完成 Android 权限集成，后端完成生产级 AI 与冲突引擎。

**前端行动项：**
- 集成 Capacitor 无障碍插件（AccessibilityService），申请前台服务权限
- 实现通知监听（NotificationListenerService）
- 实现全局复制 / 分享 Intent 接收与数据上报
- 灵动岛四态状态机与 WebSocket 事件绑定
- 完成权限引导流程 UI（首次安装引导、降级提示页）

**后端行动项：**
- RTX 5060 推理服务部署与性能调优
- BlueLM 生产 Prompt 接入（意图提取 / 歧义消解 / 冲突解释）
- 冲突检测引擎完善：时间重叠 + 出行冲突（地点距离估算）
- WebSocket Gateway 实现（reminder_triggered / conflict_detected 推送）
- 高并发场景压测（目标 QPS ≥ 100）

### Phase 3：真实环境联调与 IPC 测试（预计 1 周）

**目标：** 全链路闭环验证，真机 + 真服务联调。

**联调行动项：**
- 真机测试 Android 通知监听数据上报（不同 App 通知格式兼容）
- 端到端延迟测量：文本截取 → 后端解析 → UI 更新（Target ≤ 1.5s）
- WebSocket 断线重连与消息保序验证
- 权限申请引导流程全流程走查（安装 → 首次引导 → 降级提示）
- BlueLM 歧义消解场景测试（多义词 / 口语化表达 / 缺省时间）
- 冲突检测边界测试（连续事件 / 跨天事件 / 出行缓冲）

**IPC 测试重点：**
- 跨进程通信（App 进程与前台 Service 进程）数据序列化稳定性
- Android Intent 分享大数据包（>1MB）截断与分段处理
- 后端 WebSocket 多客户端并发推送有序性验证

## 4. Android 权限降级与异常预案

### 4.1 权限分级矩阵

| 权限 | 等级 | 申请策略 | 缺失时的降级方案 |
|------|------|----------|----------------|
| 前台服务（Foreground Service） | 必须 | 安装时说明用途 | App 无法常驻后台，强制提示开启 |
| 无障碍（AccessibilityService） | 推荐 | 首次引导弹窗 | 仅支持手动输入，无法自动嗅探 |
| 通知监听（NotificationListener） | 推荐 | 首次引导弹窗 | 降级为「分享 Intent」入口，用户手动触发 |
| 存储（存储访问框架） | 可选 | 按需申请 | 跳过截图日志功能，不影响核心流程 |

### 4.2 降级交互设计

#### 场景 1：无通知监听权限

- **灵动岛状态：** 显示「未开启通知监听」提示卡片，色彩为中性灰
- **交互引导：** 提供「打开系统设置」一键跳转按钮
- **降级入口：** 显示「分享到日程猎人」快捷入口
- **Toast 提示：**「将想要添加的日程复制，然后打开日程猎人即可自动识别」

#### 场景 2：无无障碍权限

- **灵动岛状态：** 静默态，无法自动识别用户操作
- **降级方案：** 输入界面显示「请手动输入」或「粘贴文本」入口
- **数据上报：** 手动触发时调用同一 `/api/v1/task/parse` 接口（input_type 标记为 manual）

#### 场景 3：权限部分降级（至少一种输入渠道可用）

- App 正常工作，前端以静默方式记录权限状态
- 灵动岛显示「部分嗅探功能未开启」提示，可折叠收起
- 用户可在设置页查看各权限状态并单独开启

#### 场景 4：权限全部缺失（无任何自动输入渠道）

- 强制弹出「引导开启权限」全屏页面
- 提供图文说明 + 一键开启按钮（逐个引导）
- 引导顺序：前台服务 → 通知监听 → 无障碍
- 提供「稍后再说」选项，用户可暂时跳过但每次打开 App 均会提示

### 4.3 网络异常预案

| 异常类型 | 前端处理 | 后端配合 |
|----------|----------|----------|
| 网络中断 | 本地缓存待解析文本，显示「离线同步中」状态 | — |
| 网络恢复 | 自动重试（指数退避，最大 3 次） | 幂等接口，支持重放 |
| 服务端返回 5xx | 灵动岛显示「解析服务暂不可用」提示，允许用户手动重试 | 结构化错误日志，trace_id 上报 |
| BlueLM 调用失败 | 降级为纯正则时间解析，返回 `ai_fallback: true` 标记 | Fallback Prompt 触发，记录降级日志 |
| 解析置信度过低（< 0.6） | 返回 `needs_confirmation: true`，前端弹出确认界面 | 在 `ambiguities` 字段中标注模糊字段 |

### 4.4 异常恢复策略

- **前端缓存策略：** 待解析文本存入本地 SQLite，网络恢复后按时间顺序重发
- **幂等设计：** 后端 `request_id` 相同请求直接返回缓存结果
- **降级标记：** 所有非 BlueLM 结果统一标记 `ai_fallback: true`，供后续数据清洗使用
- **用户感知：** 任何降级行为均需在灵动岛/Toast 中告知用户，不静默失败

## 5. 灵动岛状态机规范

### 5.1 四态定义

| 状态 | 英文名 | 触发条件 | 视觉特征 |
|------|--------|----------|----------|
| 静默 | Silent | 无活跃任务、无临近提醒 | 胶囊最小化、低对比度中性色 |
| 倒计时 | Countdown | 任务距离开始时间在阈值内 | 亮度递增、显示剩余时间、呼吸动画 |
| 任务提醒 | Reminder | 任务到达开始时间 | 展开半卡片、色彩强调、三段式操作 |
| 冲突预警 | Conflict Alert | 新任务触发冲突检测失败 | 红色警示、显示冲突摘要、AI 建议 |

### 5.2 状态切换规则

```
静默 ──存在临近任务──▶ 倒计时 ──进入提醒阈值──▶ 任务提醒
  ↑                              │
  │                              ↓
  └──────用户修正完成─────── 冲突预警
           │                              │
           └───────用户未处理───── 保持警示
```

### 5.3 交互规范

| 状态 | 点击行为 | 长按行为 | 手势行为 |
|------|----------|----------|----------|
| 静默 | 展开最近 3 条日程 | 快捷创建入口 | 下拉收起 |
| 倒计时 | 查看任务详情 | 跳转地图/链接 | 双击确认"已知晓" |
| 任务提醒 | 三段式操作（确认/推迟/取消） | 展开详情 | 滑动切换下一条 |
| 冲突预警 | 查看冲突链路 | AI 建议排序 | 选择处理策略 |

### 输出约束重点

- 仅输出 JSON 格式
- `confidence` 字段必须返回，范围 0.0~1.0
- 缺失信息返回 `null`，不确定时设置 `needs_confirmation=true`
- 冲突建议必须结构化、可执行，每个 `action` 必须是枚举值之一
- 建议文案控制在 50 字以内，口语化
