### ⚠️ 接口契约微差异

1. **后端 router 路径不一致**: 架构定义 POST /api/v1/tasks，后端 task.py 中路由注册为 prefix="/api/v1/task" 但额外注册了 /tasks 子路由。实际生效路径正确。
2. **GET /api/v1/tasks 请求**: 架构中 path 为 /api/v1/tasks，后端 task.py 中注册路径为 /api/v1/task/tasks（因 router prefix 为 /api/v1/task）。需修正 router 注册方式以保证路径一致。
3. **响应模型冲突**: response.py 中 ParseTaskResponseData 只包含 title/start_time/end_time/location/confidence 字段，但实际 parse 接口返回的是嵌套 data.parsed 结构 + data.is_conflict/data.conflict 等顶层字段，模型定义与实际响应不一致。

### ⚠️ 后端已知问题

1. **路由注册冲突**: task.py 中 router prefix="/api/v1/task"，但内部装饰器使用 @router.post("/tasks") 和 @router.get("/tasks")，导致实际路径为 /api/v1/task/tasks。架构要求为 /api/v1/tasks。
2. **响应模型不匹配**: ParseTaskResponseData 只有 5 个基本字段，但 parse 接口实际返回嵌套 data.parsed + data.is_conflict + data.conflict + data.suggestions 等额外字段。ConflictItem 和 Suggestion 模型字段也与实际使用不匹配。
3. **LLM 服务为模拟**: llm_service.py 返回固定模拟数据，未接入真正的 BlueLM API。
4. **Travel Conflict 简化**: 出行冲突检测未集成地图 API，仅基于时间缓冲估算。
5. **缺少提醒调度**: 后端无 reminder-service，无法主动触发提醒推送。

## 七、总结

当前项目处于「Phase 1 收尾 + Phase 2 初期」的交界状态：

- **后端骨架完整**：FastAPI 应用、路由、中间件、数据库、服务层均已实现，LLM 采用模拟模式可独立运行
- **前端链路打通**：store 已剥离 mock 依赖，vite proxy 已配置，所有请求直接发往真实后端
- **灵动岛可用**：四态 UI 就绪，可以接收真实后端响应并正确渲染
- **硬缺口明确**：Android 原生层（Capacitor/无障碍/通知监听）和 BlueLM 真实接入是两个最大的未完成模块
- **当前可运行测试**：在前端执行 `npm run dev`、后端执行 `python main.py`，即可体验真实前后端联调
